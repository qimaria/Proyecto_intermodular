⚔ Ataque
SYN Flood (DoS)
CVE-2007-6750 · hping3 --flood

El atacante envía miles de paquetes TCP SYN por segundo sin completar el handshake, saturando la tabla de conexiones semiabiertas (SYN Backlog) del servidor y agotando sus recursos de red y CPU.

🛡 Defensa
  • Activar SYN Cookies en el kernel (net.ipv4.tcp_syncookies=1) para evitar reserva de memoria por conexión incompleta.
  • Configurar UFW rate-limiting por IP para limitar conexiones simultáneas.
  • Ajustar timeouts TCP reduciendo tcp_synack_retries a 3.
  • Implementar fail2ban con jail para detectar ráfagas anómalas.
  • Filtrado de IPs con origen no válido mediante rp_filter=2.
⚠ Consecuencias

El servidor queda sin recursos para atender peticiones legítimas. La terminal del administrador deja de responder, pudiendo requerir hasta 15 minutos por comando. La página web resulta inaccesible para todos los usuarios.

Impacto alto Disponibilidad crítica Recuperación media
⚔ Ataque
Slowloris
CVE-2007-6750 · HTTP parcial lento

El atacante abre múltiples conexiones HTTP al servidor y envía cabeceras de forma extremadamente lenta, manteniendo las conexiones abiertas indefinidamente y agotando el pool de conexiones disponibles del servidor web.

🛡 Defensa
  • Configurar timeouts estrictos en Caddy: read_header 10s, read_body 30s, write 30s, idle 1m.
  • Limitar el tamaño máximo del cuerpo de petición a 1MB (max_size 1MB).
  • Bloquear métodos HTTP innecesarios; permitir solo GET y HEAD.
  • Usar UFW rate limit por conexiones entrantes al puerto 80/443.
⚠ Consecuencias

El servidor web queda incapaz de atender nuevas conexiones. Los usuarios legítimos reciben errores de timeout. El servicio permanece degradado sin consumir prácticamente CPU, lo que dificulta la detección.

Disponibilidad Detección difícil Bajo consumo CPU
⚔ Ataque
Fuerza Bruta SSH
Hydra · Ataque por diccionario

Herramientas como Hydra prueban de forma automática miles de combinaciones de usuario/contraseña contra el puerto SSH. Los bots escanean rangos de IPs buscando el puerto 22 abierto con autenticación por contraseña habilitada.

🛡 Defensa
  • Deshabilitar completamente la autenticación por contraseña (PasswordAuthentication no).
  • Usar llaves de hardware (YubiKey ed25519-sk) con PIN + toque físico (4 factores).
  • Cambiar SSH a un puerto no estándar (ej. 46217) para evitar escaneos automatizados.
  • Limitar acceso a usuarios específicos (AllowUsers alik).
  • Fail2ban bloquea IPs tras 3 intentos fallidos con espera de 30s.
  • Máximo 3 intentos de autenticación por conexión (MaxAuthTries 3).
⚠ Consecuencias

Sin defensa, el atacante puede ganar acceso root al servidor con los privilegios completos del sistema. Con las defensas implementadas (sin contraseñas + hardware key), el ataque resulta completamente inviable.

Acceso root Compromiso total Mitigado completamente
⚔ Ataque
Reconocimiento con Nmap
OSINT · nmap -sV -A -O

El atacante utiliza Nmap para mapear la infraestructura de red: descubre máquinas activas, puertos abiertos, versiones de servicios y sistema operativo. Esta fase de reconocimiento precede a cualquier ataque real.

🛡 Defensa
  • UFW bloquea todo el tráfico entrante por defecto, filtrando 997 de 1000 puertos TCP.
  • Solo puertos necesarios están abiertos: SSH (no estándar), 80, 443, 53, 67/68.
  • Puerto SSH no estándar (46217) dificulta la identificación automática del servicio.
  • El servidor NAT actúa como primera capa de aislamiento; los puertos del servidor son inaccesibles desde fuera de la red interna.
  • Nmap clasifica el SSH como "unknown" en el escaneo inicial por el puerto no estándar.
⚠ Consecuencias

El escaneo revela la superficie de ataque disponible. Gracias al hardening, solo 3 puertos son visibles desde la red interna y el SSH no se identifica inmediatamente. La exposición exterior es prácticamente nula.

Superficie mínima Info parcial al atacante Bien mitigado
⚔ Ataque
DNS Cache Poisoning
Ataque de envenenamiento DNS

El atacante inyecta registros DNS falsos en la caché del servidor DNS para redirigir el tráfico hacia IPs maliciosas. Los usuarios creen acceder al sitio legítimo mientras son dirigidos a un servidor controlado por el atacante.

🛡 Defensa
  • Activar DNSSEC en BIND9 (dnssec-validation auto) para verificar firmas criptográficas de zonas DNS.
  • Usar TSIG (Transaction SIGnature) HMAC-SHA256 entre BIND9 y Kea DDNS para autenticar actualizaciones.
  • Restringir actualizaciones de zona únicamente a la clave TSIG autorizada (allow-update { key "kea-ddns."; }).
  • Limitar consultas recursivas solo a la red interna (allow-recursion { localhost; 10.69.67.0/24; }).
⚠ Consecuencias

Sin protección, los usuarios son redirigidos silenciosamente a sitios falsos que capturan credenciales o instalan malware. Con DNSSEC + TSIG, cualquier respuesta no firmada o no autorizada es rechazada automáticamente.

Phishing masivo Robo credenciales Mitigado con DNSSEC
⚔ Ataque
Escalada de Privilegios
Privilege Escalation · Post-explotación

Una vez obtenido acceso con usuario limitado, el atacante intenta escalar privilegios hacia root mediante exploits de kernel, SUID mal configurados, sudo misconfiguration o servicios con privilegios excesivos.

🛡 Defensa
  • Root bloqueado para login directo (PermitRootLogin no); solo acceso vía sudo.
  • Usuario administrador "alik" con sudo limitado; no se trabaja habitualmente como root.
  • AppArmor activo restringe qué procesos pueden acceder a qué archivos (observado en incidencias con Kea).
  • Docker ejecuta Caddy en un contenedor aislado, limitando el daño si Caddy es comprometido.
  • Fail2ban + UFW dificultan que el atacante mantenga una sesión activa.
  • Permisos de archivos estrictos (chmod 600/700) en archivos críticos de configuración.
⚠ Consecuencias

Sin control de privilegios, un atacante con acceso a un usuario normal puede obtener control total del sistema en minutos. El modelo de mínimo privilegio + AppArmor + contenedores limita el radio de explosión de cualquier compromiso.
